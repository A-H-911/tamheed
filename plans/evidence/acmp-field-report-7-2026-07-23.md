## C. The one UNEXPECTED item: stale v1 provenance on FR-100/FR-107 (live-side, pre-existing)

The live rows' `custom_attributes.v1` still hold the **2.4.0-era escaped-pipe shear**:

- live: `"Requirement": "…(Superseded \\", "Priority": "Deprecated). No backward…", "Source": "S", …` (columns shifted)
- scratch: `"Requirement": "…(Superseded | Deprecated). No backward transitions…", "Priority": "S", …` (correct)

Root cause chain: the 2.4.0 hand-repair fixed the **visible columns** (title/statement/priority)
but deliberately never sent `custom_attributes` on update — the G1 guard (an update REPLACES the
whole JSON), so the corrupted snapshot survived. findings_6's claim that requirements were
"absent from the diff entirely — byte-exact" was **overstated**: its diff script did not compare
`custom_attributes`. The 2.5.x parser has produced a clean snapshot since 2.5.0; only the live
rows are stale. **REPAIRED same day by operator order (PE-130):** full-row
`entity_upsert` of FR-100/FR-107 including the corrected `custom_attributes` — here the
whole-JSON replace is the *desired* behavior. One tool note: `custom_attributes` must be sent
as a **JSON string**; a JSON object fails binding ("type 'dict' is not supported") and rolls
the batch back — worth either accepting dicts or saying so in the tool description. Also a small
**method lesson for runbook §8**: the field diff must enumerate the union of columns including
JSON blobs, or it silently under-reports exactly the way findings_6 did.
