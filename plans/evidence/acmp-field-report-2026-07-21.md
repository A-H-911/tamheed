# Tamheed v1→v2 migration — field report / feedback

**From:** a real-world `migrate`-mode run against a production Keystone v1 package
**Package:** ACMP (`docs/`) — v1 validator 7/7 green; 218 requirements, 74 ACs, 74 audit verdicts,
33 ADRs, 28 decisions, 58 OQs, 155 WBS items, 1,032 trace edges; manifest generated by Keystone
itself (2026-07-06), grown under `resume`/execution since.
**Environment:** Windows 11 Pro, Claude Code (plugin install `tamheed@tamheed` 1.0.0), `uv 0.9.21`,
Python 3.13.7.
**Outcome:** migration succeeded and `gate_run` reached `ready: true` — but only after two blockers
were worked around outside the MCP transport and several silent-loss gaps were backfilled by hand.
Everything below is from direct source inspection (`server/migrate.py`, `tamheed_server.py`,
`db/schema.sql`, `scripts/validate_package.py`) plus the actual run; line numbers refer to 1.0.0.

---

## A. Blockers (vanilla migration of this package cannot succeed)

### A1. MCP server deadlocks on Windows when pre-flight spawns the validator subprocess
`preflight()` runs `subprocess.run([sys.executable, VALIDATOR, source, "--json"])`
(`migrate.py:148`). Inside the stdio-hosted MCP server on Windows (launched via `uv run` per the
bundled `.mcp.json`), the child python **freezes at process startup — 0 CPU, tiny working set —
indefinitely**. Reproduced twice. The uv venv's `Scripts\python.exe` trampoline spawns a child
interpreter; both processes in the pair sit frozen. The same validator invocation from a normal
shell completes in ~2 s, so it is specific to spawning from the MCP server process.

- Killing the child pair unblocks the handler, which then reports
  `v1 validator failed (exit 4294967295)` with an empty report — technically loud, but the operator
  has no way to see it was a spawn hang rather than a validator failure.
- **Suggested fix:** don't shell out at all. `migrate.py` already imports the frozen validator as a
  library (`import validate_package as vp`, line 21) for parsing; run the gates in-process too.
  If the subprocess must stay, add `stdin=subprocess.DEVNULL`, a timeout, and (Windows)
  `creationflags=CREATE_NO_WINDOW`, and surface a distinct "pre-flight spawn timed out" error.
- Related DX: first launch also pays uv's cold PEP-723 resolve, so a wedged call and a slow cold
  start are indistinguishable from the client. A trivial `ping`/`server_info` tool (which could also
  report the resolved package root — see A3) would make this diagnosable.

**Workaround used:** drove the plain handler functions in-process
(`sys.path.insert(server dir); import tamheed_server as ts; ts.PACKAGE_ROOT = Path(<repo>)`) —
worked flawlessly; the handlers being transport-free plain functions is a genuinely good design.

### A2. `decisions.promoted_to` FK + naive cell parsing aborts the whole populate
The parser takes the **first governed ID of any type** from the "Promoted to" cell
(`migrate.py:207`), but the column is `promoted_to TEXT REFERENCES adrs(id)` (`schema.sql:179`).
Our DEC-028 cell reads `n/a (scheduling; amends **DEC-019**'s phase, see **D-11**, **ADR-0006**
unchanged)` → parser stores `DEC-019` → `sqlite3.IntegrityError: FOREIGN KEY constraint failed`
and the **entire one-transaction populate dies**, with no indication of which table/row was at
fault (the exception propagates raw out of the batch loop, `migrate.py:512-536`).

- **Suggested fixes:** (1) only accept `ADR-*` tokens for `promoted_to`, else null it and add an
  `unmapped` note; (2) wrap populate's insert loop so a constraint failure reports
  `table/id/constraint` to the operator — a migration of ~2,000 rows failing with a bare
  IntegrityError is very expensive to root-cause (we had to bisect with a scratch store).

### A3. Package root / source dir resolve against the server's cwd
The bundled `.mcp.json` passes `--package-dir "."` and the docs default matches. Claude Code's own
plugin docs say a stdio server's working directory is *not* guaranteed and recommend
`${CLAUDE_PROJECT_DIR}` (exported to plugin server processes). A relative `source_dir` has the same
issue. Nothing in the flow verifies where the package actually landed.
**Suggested fix:** default `--package-dir` to `CLAUDE_PROJECT_DIR` when set (or use
`${CLAUDE_PROJECT_DIR}` in the bundled `.mcp.json` args), and echo the resolved absolute root in
every `package_create/open/migrate` result.

## B. Silent data loss (migration "succeeds", operationally live artifacts vanish)

### B1. ADR files without YAML front-matter migrate as **zero ADRs** — and nothing flags it
`_map_adr` requires `front_matter.id` (`migrate.py:289-291`). Our ADRs — generated by Keystone
itself — use the common MADR shape: `# ADR-0031: title` heading + `- Status: Accepted` bullet, no
YAML. Result: **0 of 33 ADRs migrate.** Each file gets one `unmapped` line, but:
- the fidelity check cannot catch it — ADR ids never enter `plan.defined`, so `identifier_gaps`
  stays empty; only a count delta (23→0) hints at it, easily misread next to legitimate deltas;
- the weak-definition synthesizer has no ADR branch (prefix list at `migrate.py:452` omits
  ADR/EXP/POC), so even the heading-level definitions produce nothing;
- combined with A2, missing ADRs also break any decision that was promoted (FK target absent).
**Suggested fix:** fall back to heading + `- Status:` parsing (id from `# (ADR-\d+)`), which also
matches what Keystone v1 emitted. At minimum: refuse to proceed (or require explicit
acknowledgment) when a whole family parses to zero against a nonzero manifest count.

### B2. Build slices don't exist in the mapping at all
The v2 schema is explicit that slices are what branches/PRs/ACs bind to, yet `migrate.py` never
creates a `slice` row (nothing targets the `slices` table; `INSERT_ORDER` doesn't even list it).
Our P1–P19 ladder — the project's actual execution scheme — is invisible: single-letter `P17`
tokens don't match `ID_TOKEN_RE` (`[A-Z]{2,5}-\d`), and `planning/roadmap.md` matches no
`NARRATIVE_KINDS` entry, so the ladder tables and prose are dropped wholesale. Not listed in
`unmapped` (which is id-level only).
**Suggested fix:** even if v1 had no governed slice prefix, migrate the roadmap as a narrative doc
(add `planning/roadmap` to `NARRATIVE_KINDS`) so the ladder text survives; document that slices
need manual backfill, or offer a mapping hook.

### B3. Deferred-work register dropped silently and completely
`deferred_work` is in `INSERT_ORDER` (`migrate.py:479-483`) but no code path ever adds rows.
Our `D-01…D-23` (ungoverned `D-` prefix, per Keystone's own conventions) are invisible to the
ID regex and the file matches no narrative kind → 23 live deferred-work items vanish with no trace
in `unmapped`. Ironic given `schema.sql:459` calls deferred work "the strongest signal".
**Suggested fix:** recognize `execution/deferred-work-register.md` (the standard v1 filename) and
map `| D-nn |` rows to `DW-` ids; or at least migrate the file as narrative.

### B4. All AC statements truncated to 120 characters
The AC register's statement column is headed **"Given / When / Then"** (Keystone's own format),
which is not in the title-alias list (`migrate.py:170-173`); the fallback `_clean_line` caps at 120
chars (`migrate.py:111-113`) and the AC branch writes that truncated string into **both** `title`
and `statement` (`migrate.py:250-253`). Every one of our 74 ACs exceeds 120 chars (min 167,
max 443) — all were gutted; full text survived only inside the `custom_attributes` v1 blob.
**Suggested fix:** add `given / when / then` (and `criterion`) to the alias list, and never let
`statement` inherit the title cap — statement should be the raw cell.

### B5. All audit verdicts land evidence-less ("narrated")
The audit parser reads only a column literally named `evidence` (`migrate.py:374`); Keystone's
audit table uses **"Test ref"** (plus Section/Notes). Result: 74/74 verdicts migrate with
`evidence NULL`, and `gate_run` then reports "narrated verdicts are the graded party grading
itself" — an indictment manufactured by the migration itself. Section/Notes are preserved nowhere
(audit inserts carry no `custom_attributes`, `migrate.py:521-525`).
**Suggested fix:** accept `test ref` as an evidence alias; carry the remaining audit columns in
`custom_attributes` like every register row does.

### B6. Smaller unrecoverable losses
- **Profile string destroyed:** `_norm_profile("software-platform (on-prem, ≤20 users)")` →
  `unknown`, and `"profile"` is in the extras-exclusion list (`migrate.py:333-335`) so the original
  string is stored **nowhere**. Preserve the raw value in `custom_attributes`.
- **`generated` vs `generated_at`:** our Keystone-emitted manifest uses `generated` →
  `created_at="v1-migration"` (`migrate.py:346`). Accept both spellings (the migration contract
  already promises manifest-as-found tolerance).
- **Omission keyword gaps:** `research/research-plan.md`'s omission reason matches no
  `OMISSION_KEYWORDS` entry and is silently dropped; the `planning/milestones.md` omission is
  recorded even though 6 MS rows migrate (family shows both present and omitted).
- **Narrative coverage is ~20% and the preview doesn't say so:** only charter/exec-summary/
  architecture/tech-comparison/READMEs/governance migrate as narrative docs; all ~35 `domain/*.md`,
  test-strategy, work-breakdown prose, checkpoints, definition-of-done, glossary are dropped except
  governed table rows inside them. `unmapped` lists id-level issues only — **dropped files are
  never listed**. A "files not migrated" section in the preview would make the coverage honest.

## C. Gate-integrity issues (green that shouldn't be trusted)

### C1. G-TRACE passes vacuously after migration
`mvp=1` requires the literal substring `"mvp"` in the Priority/Scope cell (`migrate.py:181`).
Keystone v1 uses MoSCoW `M/S/C/W` → every requirement migrates `mvp=0` → `g_trace_failures`
filters `mvp=1` and returns zero rows. The post-flight fidelity and `gate_run` both look green
while the traceability guarantee has evaporated. **Suggested fix:** map MoSCoW (`M`/`Must` → mvp)
or, better, flag "0 MVP requirements after migration" as its own warning — it is almost never true.

### C2. G-COMPLETE's placeholder regex flags literal code tokens — including inside the
preservation mechanism itself
`\{\{[^}]*\}\}` (`tamheed_server.py:116-118`) fired on a deferred-work title quoting JSX
`style={{}}`. Rewording the title moved the failure into `custom_attributes`, because the scan
covers **all** TEXT columns — i.e. the field whose whole purpose is verbatim v1 preservation trips
the gate for being faithful. We had to scrub the preserved text. **Suggested fix:** exempt
`custom_attributes` from the placeholder scan (it is provenance, not authored content), or skip
matches inside backticked code spans.

### C3. Preview vs fidelity asymmetry
Count deltas are only computed in the post-confirm fidelity report (`migrate.py:556-564`); the
preview shows `manifest_counts` beside parsed counts but computes nothing, and edge-endpoint
`unmapped` entries are appended only during populate (`migrate.py:533`). The operator confirm
(stage 3) is therefore taken on less information than the tool already has. Compute deltas in the
preview.

## D. Design friction (arguably working as intended, but worth a documented path)

### D1. Immutability triggers vs migration repair
Approved ACs and recorded verdicts refuse UPDATE ("approved acceptance criteria are immutable:
supersede, never edit"). Correct governance — but it means every repair of a migration gap
(B4, B5) **must be injected into the plan before populate**; post-hoc `entity_upsert` is
structurally impossible, and re-keying via supersession would break the governed AC ids that code,
CI, and audits reference. There is no supported hook for this. We ended up mutating `plan.rows` /
`plan.audits` between `parse_v1()` and `populate()` in a driver script. **Suggested fix:** a
documented enrichment hook (e.g. `package_migrate(..., patch=<file>)` applied to the parsed plan,
reported in the preview) — or bless the parse→patch→populate sequence as the official repair path.

### D2. `entity_upsert` partial rows fail on existing entities
The upsert is `INSERT ... ON CONFLICT(id) DO UPDATE` (`tamheed_server.py:253-257`), so the INSERT
half still evaluates NOT NULL on omitted columns — a partial `{id, statement}` update of an
existing row fails `title NOT NULL` before conflict resolution is reached. Callers must send full
rows. Worth a docstring note (or an explicit `entity_patch`).

### D3. Failed populate leaves a poison directory
`PackageStore.__enter__` mkdirs `<pkg>/data` before any insert; after a rollback the empty
directory survives and every retry refuses with "destination package already exists"
(`migrate.py:493-495`). Clean up the created directory when populate fails.

### D4. `_map_adr` section-order trap
`next(v for k,v in sections.items() if "decision" in k.lower())` matches "Decision **Drivers**"
before "Decision **Outcome**" in MADR-style files — the `decision` column would get the drivers
list. Prefer "decision outcome", exclude "driver". (Latent today only because B1 prevents ADRs
from reaching this code with MADR files at all.)

## E. What worked well (keep these)

- **Staged preview → operator confirm → populate** held its contract exactly; preview is genuinely
  write-free.
- **One-transaction atomicity:** every failed populate left no partial package (modulo D3's empty
  dir).
- **Per-item verdicts on `entity_upsert`** name the violated constraint precisely — the
  immutability rejections were instantly understandable.
- **Handlers as plain functions** (no transport coupling) is what made recovery from A1 possible
  at all. Please preserve this property.
- **Canonical JSONL determinism and relocatability:** no absolute paths in the data; byte-stable
  output; `.lock` lifecycle clean (released on close, loud on contention).
- **The frozen v1 validator** is faithful to Keystone 1.0.0 semantics — including the
  acceptance-audit duplicate-ID skip that older validators got wrong.
- **`review.html`**: self-contained, CSP'd, zero-JS, deterministic — committed well, diffs cleanly.
- The v1 quirk-absorption that *is* implemented (manifest `package|project`, extras preservation,
  weak definitions, `custom_attributes` for every unmapped cell) is the right philosophy — the
  gaps above are places where the philosophy isn't applied uniformly, not a wrong design.
